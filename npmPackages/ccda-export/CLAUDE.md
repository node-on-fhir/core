# CLAUDE.md — @node-on-fhir/ccda-export

Migrated from Atmosphere `clinical:ccda-export` (2026-06-14, MIT). C-CDA export + clinical document management for ONC §170.315(b)(1) (Transitions of Care). Two routes — `/ccda-export` and `/clinical-documents` (both render `CCDAExportPage`, the second in list view, `requireAuth`).

**Not actually gated.** The old package.js's `Package.onTest` block `api.use('clinical:clinical-documents')` — that's a **stale reference to the package's own former name** (the file header still says `packages/clinical-documents/package.js`; the package was renamed to `clinical:ccda-export`). Test-only, never ES-imported. Real onUse deps are app infra (`mongo`, `react-meteor-data`, `clinical:hl7-resource-datatypes`, `matb33:collection-hooks` weak) — all present. Same false-gate pattern as accounts-management / consent-generator / vital-signs.

**Structure:** `client/pages/{CCDAExportPage,ClinicalDocumentsList,ClinicalDocumentDetail}.jsx`, `lib/collections/{ClinicalDocuments,DocumentRevisions}.js` (isomorphic `Mongo.Collection`s, already ES-exported — no bare-globals), `server/publications/{clinicalDocuments,documentRevisions}.js`, `server/methods.js`. No collection-name collision with the host app. No external npm deps, no old-MUI, no Atmosphere-isms, no `meteor/http`. Collections re-exported from `client.js` and via subpath exports (`./lib/collections/*`). client.js preserves the full index.jsx (ClinicianWorkflows / SidebarElements / FooterButtons / ModuleConfig / settings gates). Server exposes no Package-registry symbols → `Package['@node-on-fhir/ccda-export'] = {}` (harmless). Monorepo-tracked → fresh git init (nested repo).
